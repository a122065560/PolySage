================================================================================
一、项目概述
================================================================================

RoundTableAI 是一款 macOS 原生桌面应用，让多个 AI 大模型在同一个"圆桌"上
自动轮转讨论同一个话题，由"军师"统筹结案，全程无需任何 API Key。

核心特点:
  - 零 API Key: 通过 Playwright 直接控制浏览器网页版 AI 进行操作
  - 多 AI 协作: 支持 5 个内置中文 AI 平台（DeepSeek、智谱清言、通义千问、
                MiniMax、Kimi）+ LM Studio 本地模型
  - 军师+谋士架构: 指定一个 AI 为军师统筹全局，其余为谋士各抒己见
  - 自动轮转讨论: Phase 0-4 四阶段讨论流程，军师主导，谋士并行回复
  - 三级线程架构: UI线程 → 大脑线程 → AIWorker，确保UI流畅+AI操作并行
  - 主公密令: 讨论中用户可随时发多条密令，累积后合并发给军师
  - 思考内容过滤: 自动识别并剥离 AI 的思考过程，只提取实际发言
  - 容错机制: 连续失败自动禁用，asyncio 异常保护

三、三级线程架构（核心设计）
================================================================================

整个系统采用三级线程架构，这是项目的核心设计，确保 UI 流畅、AI 操作并行、
讨论调度有序。

┌─────────────────────────────────────────────────────────────────────┐
│ 1级 主线程 (Qt UI)                                                  │
│                                                                     │
│ PyQt6 主窗口。只负责接收信号、更新 UI、响应用户操作。               │
│ 不执行任何耗时操作，保证界面永远流畅。                              │
│                                                                     │
│ 通信方式: pyqtSignal 信号 ↓                                        │
├─────────────────────────────────────────────────────────────────────┤
│ 2级 WorkerThread (大脑)                                             │
│                                                                     │
│ 独立 asyncio 事件循环。管理 ChromeManager、调度所有 AIWorker、      │
│ 执行讨论主循环（HostedMode）。                                      │
│                                                                     │
│ 职责:                                                               │
│   - 统筹和运转所有 AIWorker                                        │
│   - 接收3级线程的所有报告（状态、回复、发言状态等）                 │
│   - 多条消息统筹：同一AI发了多条时合并到一起                        │
│   - 主公密令合并：累积队列中的密令，合并发给军师                    │
│   - 二次提取：发送后等待2秒再次提取，确保内容完整                   │
│   - 思考过滤：检测到思考内容时重新提取实际回复                      │
│                                                                     │
│ 通信方式: asyncio.Queue ↓                                          │
├─────────────────────────────────────────────────────────────────────┤
│ 3级 AIWorker (每个AI一个 asyncio Task)                              │
│                                                                     │
│ 每个 AI 平台一个独立 asyncio Task。只关注自己的 AI 页面。           │
│                                                                     │
│ 职责:                                                               │
│   - 监控AI是否在发言（吐token）                                    │
│   - AI发言完毕后，提取内容传输回大脑                                │
│   - 如果一次没截全，做二次提取，保证无遗漏                          │
│   - 不做任何判断（判断交给大脑）                                    │
│   - 执行大脑的 send 命令                                            │
│   - 监控登录状态和思考模式状态                                      │
└─────────────────────────────────────────────────────────────────────┘

为什么三级而非两级?

  1. 主线程不能阻塞: PyQt6 的 UI 渲染在主线程，任何 await 都会卡死界面
  2. 大脑需要独立循环: HostedMode 的讨论主循环是 async 的，必须在独立
     asyncio 循环中运行
  3. 每个 AI 独立隔离: 一个 AI 崩溃不影响其他 AI，大脑可以感知并跳过

3级 AIWorker 报告类型:

  ("status", name, color, msg)         — 登录/思考模式状态报告
  ("page_opened", name, "")            — 页面打开成功
  ("page_failed", name, "原因")        — 页面打开失败
  ("ai_speaking", name, True/False)    — AI开始发言/发言完毕
  ("reply", name, text)                — 完整回复
  ("reply_fragment", name, text)       — 回复片段（可能不完整）
  ("reply_error", name, error_msg)     — 回复错误
  ("error", name, error_msg)           — 通用错误

================================================================================
五、讨论流程（Phase 0-4 完整描述）
================================================================================

讨论由 HostedMode.run() 主导，分为 5 个阶段:

  [Phase 0] 文件上传（并行）
    如果用户拖入了文件，并行上传到所有 AI 页面。
    支持 .txt / .md / .csv 格式。
    验证文件名出现在页面上确认上传成功。
         │
         ▼
  [Phase 1] 开场白确认（并行）
    并行向所有 AI 发送开场白（含角色设定、规则、哨兵标记说明），
    等待每个 AI 回复 <ok> 确认。
    fast_wait=True，timeout=30s，软超时 180s，超时跳过。
         │
         ▼
  [Phase 2] 军师发言
    向军师发送讨论话题和上一轮讨论记录:
      第一轮: build_arbiter_first_prompt（话题+谋士名单）
      后续轮: build_arbiter_round_prompt（上一轮记录+分歧焦点）
    军师先发言给出方向性意见。
         │
         ▼
  [Phase 3] 大脑转发 + 谋士并行回复
    大脑将军师发言转发给所有谋士:
      build_strategist_round_prompt（军师发言+上一轮讨论+分歧焦点）
    谋士并行回复（asyncio.as_completed，先回复的先展示）。
    软超时 180s，超时的谋士本轮跳过（消息中标注AI名称）。
         │
         ▼
  [Phase 4] 轮转 / 结案
    大脑收集所有回复 → 检测结案信号 → 军师下一轮发言 → 循环 Phase 2-4
    直到:
      ① 军师发出 <结案> 标识 → 立即结束
      ② 所有 AI 都说了 <End> → 军师整合最终方案
      ③ 达到最大轮数（默认 100）→ 强制结案

每轮循环开始前，会先处理主公密令队列（详见第七节）。


================================================================================
六、军师结案机制
================================================================================

结案规则:
  1. 军师发出 <结案> 标识 → 立即停止讨论，军师回复即为最终方案
  2. 所有 AI 都说了 <End> → 军师整合所有谋士意见，输出最终方案
  3. 达到最大轮数（max_rounds=100）→ 强制军师结案

防过早结案:
  前 min_rounds_before_arbitration（默认 3）轮内不允许结案。

军师可选模式 (arbitrator 配置项):
  - auto: 自动提取 <结案> 标记前的内容
  - lm_studio: 调用 LM Studio 本地模型汇总历史记录
  - 指定AI名: 发给该 AI 汇总

结案后:
  - 返回 {"history", "final_result", "ended_by", "rounds"}
  - UI 展示最终方案
  - 用户可继续追问（continue_discussion 模式）


================================================================================
七、主公密令队列（多条累积+合并发送）
================================================================================

设计目的:
  用户在讨论进行中可随时发送多条消息（密令），系统累积后合并发给军师，
  避免打扰军师吐 token。军师不直接回复密令，而是将密令纳入下一轮上下文。

实现:

  1. 用户发送:
     HostedModeTab._on_send() 检测到 hosted.is_running() == True
     → 调用 hosted.submit_user_input(text)
     → 消息加入 _user_input_queue（asyncio.Queue，线程安全）
     → UI 提示"主公密令已暂存（队列中N条），军师空闲时自动发送"

  2. 累积与合并（每轮循环开始前，军师上一轮已回复完毕时）:
     all_user_msgs = []
     while not self._user_input_queue.empty():
         all_user_msgs.append(self._user_input_queue.get_nowait())

     合并逻辑:
       1条 → 直接用 combined_msg = all_user_msgs[0]
       多条 → combined_msg = "\n\n".join(
                f"密令{i+1}: {msg}" for i, msg in enumerate(all_user_msgs)
              )

  3. 发送:
     用 build_supplement_prompt(combined_msg, arb_name, end_signal) 构造提示
     只发给军师
     军师将密令纳入下一轮上下文统筹

开场白规则中明确告知 AI:
  "如果主公中途发密令，则只发给军师，军师不用回复主公，
   只是把密令需求纳入统一考量，在后续讨论中引导谋士一并考虑"


================================================================================
八、浏览器控制 ChromeManager（5级发送+轮询验证+三层过滤提取）
================================================================================

这是项目最复杂的模块（127KB），负责所有与 AI 网页的交互。

8.1 Chrome 启动
--------------------------------------------------------------------------------

通过 macOS LaunchAgent 以调试模式启动 Chrome:
  --remote-debugging-port=9222
  用户数据目录: ~/.roundtableai/chrome-data（与用户日常 Chrome 隔离）

关闭时:
  1. launchctl unload（正常关闭）
  2. 如果端口仍被占用 → pkill -f "remote-debugging-port=9222"（强制关闭）

8.2 消息发送（5级发送策略）
--------------------------------------------------------------------------------

send_and_wait() 是核心方法，_click_send_button() 采用 5 级发送策略依次尝试:

  [策略1] 键盘发送
    A: JS dispatch keydown Enter
    A2: 直接点击智谱清言专用 div.enter.is-main-chat
    B: page.keyboard.press("Enter")
    C: Meta+Enter
    D: Ctrl+Enter

  [策略2] 配置选择器
    使用 AI 平台配置中指定的发送按钮 CSS 选择器

  [策略3] 输入框附近查找
    ~20种通用选择器（div[class*='send']、div.enter-icon-container 等）

  [策略4] aria-label 精确匹配
    button[aria-label='发送'] / button[aria-label='Send']

  [策略5] JS 全局查找
    遍历所有 button/div/span/img，按优先级匹配 send/submit/enter 类名

每个策略后都调用 _verify_sent() 验证是否发送成功。

8.3 发送验证（轮询检测）
--------------------------------------------------------------------------------

_verify_sent() 采用 3 秒轮询（500ms × 6 次），解决 React 异步渲染导致
DOM 延迟更新的问题:

  - 检测搜索弹窗 → 按 Escape 关闭，返回失败
  - 轮询6次:
    * 输入框元素脱离 DOM（detached）→ 发送成功
    * 输入框已清空或内容变化 → 发送成功
    * 页面内容增长 > 50 字符 → 发送成功

8.4 回复提取（三层过滤）
--------------------------------------------------------------------------------

从 AI 网页提取最新回复时，采用三层过滤排除非 AI 回复内容:

  第一层: JS 智能提取器（_extract_last_response）
    注入大段 JS 到页面，包含多个检测函数:
    - isThinkingElement(el): class 含 think/reasoning/chain-of-thought 等
    - isUIElement(el): button/a/input/role=button/toolbar/action
    - isSidebarContent(el): 超过一半行是 UI 标签且 ≥3 行
    - isAdContent(el): GLM-+旗舰模型、扫描二维码、保存名片等
    - isFileAttachment(el): 正则匹配文件名+大小（如"文件.txt 36KB"）
    - isPlaceholderText(el): Kimi 占位符（"问点难的""算力不足"等）
    - isUILabel(el): UI_LABEL_BLOCKLIST 黑名单匹配
    - isUIReplyLabel(el): "回复N/M"分段标签
    - isInitPrompt(el, markers): 开场白特征词
    - isSentMessage(el, sentMsg): 工具发送的消息
    - isSentPromptBlock(el): "【第N轮 - 军师/谋士发言】"开头
    - isUserMessage(el): data-role="user"、class含user/me/human

    提取流程:
    9个 containerSelectors 查找 → 逐个过滤 → 去重 → 按位置排序
    → 取 replyCountBefore 之后的新回复（优先最长）→ 回退选择器

  第二层: Python 验证（send_and_wait，max_retries=10）
    重试循环中多重验证:
    - 初始化提示检测
    - 占位符检测
    - 广告内容检测
    - 侧边栏内容检测
    - 分段UI标签检测（"回复N/M"）
    - 文件附件标签检测（正则匹配）
    - 太短检测（<20字 → 重试）
    - 发送消息本身检测
    - 与上次回复相同检测
    - 内容增长但提取过短检测
    - 截断检测（不以标点结尾 → AI可能还在输出）

  第三层: 重试机制
    智能重试间隔:
    - 页面变化 → 正常等待（retry_interval 秒）
    - 页面未变化 → 缩短至 3 秒
    - 超过总超时 → 停止重试

8.5 回复完成检测
--------------------------------------------------------------------------------

_wait_for_reply_complete():
  - 监测页面内容增长
  - 连续 4 秒无变化 → 判定完成
  - 内容缩小（思考过程折叠）→ 额外等待


================================================================================
九、思考内容检测与过滤
================================================================================

9.1 思考内容检测 _is_thinking_content(text)
--------------------------------------------------------------------------------

检查前 8 行是否含 ≥3 个思考关键词:

  中文关键词: 分析输入、构思回复、起草回复、对照约束、思考过程、
              深度思考、最终润色
  英文关键词: Thought、Analysis、Reasoning
  智谱清言格式: Role:、Goal:、Strategy:、Current State:、
                Drafting the Response、Key Points to Make、
                Response Structure、Summary of Conflict、
                Analysis of Disagreements、Phase 1: ~ Phase 4:、
                Action:、Direct the next round

9.2 思考内容剥离 _strip_thinking_content(text, ai_name)
--------------------------------------------------------------------------------

5 级策略定位实际回复起始点:

  策略1: 查找【开头的行（中文方括号，通常是正式回复标题）
  策略2: 查找"一、二、三、"中文序号开头
  策略3: 查找AI名称开头的正式回复行
  策略4: 查找"各位谋士"/"各位同僚"/"主公"开头的行
  策略5: 行级过滤（移除思考标记段落，处理列表项）

找到回复起始行后，截取该行及之后的所有内容作为实际回复。

9.3 UI 元素过滤 _strip_ui_elements(text, ai_name)
--------------------------------------------------------------------------------

过滤网页 UI 元素文字行:
  AI编辑、收藏至知识库、收藏、编辑、复制、分享、点赞、重新生成、
  停止生成、继续生成、Copy、Edit、Share、Regenerate

保留 >30 字的行（可能是包含这些词的长段落）。

9.4 大脑二次提取+思考过滤（HostedMode._send_to）
--------------------------------------------------------------------------------

_send_to() 在 send_and_wait() 返回后执行大脑后处理:

  1. 二次提取: 等待2秒后调用 _extract_last_response()
     若二次提取更长 → 使用二次结果（"一次没截全"）

  2. 思考内容检测: _is_thinking_content(reply)
     为真时:
       a. 先用 _strip_thinking_content() 尝试提取实际回复
       b. 若过滤后仍太短 → 二次提取原始内容 + 再次 _strip_thinking_content
       c. 若仍无法提取 → 使用当前内容

  3. 通知UI: ai_speaking=True/False


================================================================================
十、AI 容错机制
================================================================================

10.1 连续失败自动禁用
--------------------------------------------------------------------------------

  谋士: ai_fail_count{name: count}
    - 连续 3 次超时/失败 → 加入 ai_disabled 集合
    - 本次讨论不再发送消息给它
    - 其他 AI 正常讨论不受影响
    - 回复成功时重置计数器

  军师: consecutive_failures
    - 连续 5 次失败 → 讨论结束
    - 失败后等待 15 秒重试

  无谋士回复:
    - 连续 3 轮无任何谋士回复 → 结束讨论

10.2 页面失效自动重建
--------------------------------------------------------------------------------

  _send_to() max_retries=3:
    - 检测 page.url 失效 → 重建页面
    - 等待 5 秒页面加载
    - 重试发送
    - 3次都失败 → 返回错误

10.3 asyncio 异常保护
--------------------------------------------------------------------------------

  三层异常钩子:
    1. sys.excepthook — 主线程同步异常
    2. threading.excepthook — 子线程异常（WorkerThread）
    3. loop.set_exception_handler — asyncio 异常
       不重新抛出，防止进程崩溃，记录到异常日志

  _brain_loop 用 try/except 包裹，异常只 log_warning 不中断。

================================================================================
十七、关键设计决策
================================================================================

17.1 为什么不用 API Key？
--------------------------------------------------------------------------------

大多数中文 AI 平台的免费网页版功能完整但 API 收费。通过 Playwright 控制
浏览器直接操作网页版，零成本使用全部功能（包括深度思考、文件上传等高级特性）。

17.2 为什么用 qasync 而非纯 asyncio？
--------------------------------------------------------------------------------

PyQt6 的信号槽机制必须在 Qt 事件循环中运行，而 Playwright 的所有操作都是
async 的。qasync 将 asyncio 循环集成到 Qt 循环中，使 async def 槽函数中的
await 不阻塞 UI 渲染。

17.3 为什么三级线程而非两级？
--------------------------------------------------------------------------------

两级架构（UI + 工作线程）下，讨论主循环和工作线程共用同一个 asyncio 循环，
一个 AI 的 Playwright 操作异常可能导致整个循环崩溃。三级架构下，每个 AIWorker
是独立的 asyncio Task，崩溃后大脑线程可以感知并跳过，不影响其他 AI。

17.4 为什么用哨兵标记而非 NLP 判断？
--------------------------------------------------------------------------------

哨兵标记（<ok> / <End> / <结案>）是 100% 可靠的状态判断方式。开场白中明确
告知 AI 规则，AI 只需在回复中包含标记即可。不依赖 NLP 语义分析，零误判。

17.5 为什么主公密令要累积合并？
--------------------------------------------------------------------------------

军师在吐 token 时不能打扰（可能导致页面状态混乱）。用户发的多条密令先累积
在队列中，等军师上一轮回复完毕（空闲）时，合并为一条发给军师。军师不直接
回复密令，而是将密令需求纳入下一轮上下文统筹。

17.6 为什么需要二次提取？
--------------------------------------------------------------------------------

AI 网页使用 React 等框架异步渲染，首次提取时 AI 可能还在输出中，导致内容
不完整。等待 2 秒后再次提取，取更完整的版本。如果两次提取差异大，说明
页面状态不稳定，需要进一步处理。

17.7 为什么需要思考内容过滤？
--------------------------------------------------------------------------------

部分 AI 平台（如智谱清言）开启了深度思考模式后，会在回复中先输出思考过程
（如 "Role: 军师\nGoal: ...\nStrategy:..."），然后才输出实际回复。如果不过滤，
思考内容会被当成"军师回复"发给其他 AI，导致讨论混乱。通过 _is_thinking_content
检测 + _strip_thinking_content 剥离，确保只提取实际发言。
